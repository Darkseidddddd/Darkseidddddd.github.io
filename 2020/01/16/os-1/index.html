多进程与多线程
发表于 2020-01-16| 更新于 2020-02-11|操作系统

一. 实验目的

​ 通过完成下述工作了解LINUX的安装，系统的基本命令，进程的创建及进程间的通信及同步与互斥。

二. 实验内容

  1. 下载并安装LINUX，可以使用Ubuntu或其它的版本。熟悉UNIX/LINUX的常用基本命令如ls、who、wc、pwd、ps、pstree、top,cat,cd,chgrp,chmod,chown,comm,cmp,cp,rm,diff,mv,rmdir等，了解环境。
    • 比较 fork() 和 clone() 的功能：利用 fork() 生成子进程（验证变量之间不能共享）。
    • 用 clone() 生成线程（设置CLONE_FS，CLONE_VM，CLONE_FILES）并验证其共享。
    • 利用pthread库实现多线程的的归并排序算法（多个线程完成一个列表中的部分数据排序，再由另外一个线程完成其合并得到有序的表）。
  3. 利用pthread库，通过其中的互斥锁和信号量来实现有限缓存的生产者-消费者问题。
  4. 使用System V的信号灯，实现第二类读者-写者问题（写者优先，一旦有写进程申请时后面到的读者不能立刻进入临界区进行读操作）。

三. 所遇问题及解决方法

1. Segmentation fault (core dumped)

产生原因：

Segmentation fault 段错误。

Core Dump 核心转储（是操作系统在进程收到某些信号而终止运行时，将此时进程地址空间的内容以及有关进程状态的其他信息写出的一个磁盘文件。这种信息往往用于调试），其实“吐核”这个词形容的很恰当，就是核心内存吐出来。

出现这种错误可能的原因（其实就是访问了内存中不应该访问的东西）：

3，多线程读写的数据未加锁保护。

对于会被多个线程同时访问的全局数据，应该注意加锁保护，否则很容易造成核心转储。

2. 多线程执行速度问题

第三个归并排序，我在租借云服务器上跑，用一个线程的时候，跑200000的数据要40多毫秒，而用多线程的时候，需要60多毫秒，线程并没有起到加速的作用。后来发现是因为我的服务器上是单个CPU，且是单核的，所以并不能严格产生并行线程，所以增加线程数的时候，创建线程会导致耗费更多的时间。
